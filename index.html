01
URL Shortener (TinyURL)
Generate short codes → redirect to originals at scale
Hashing Cache DB ▶
Architecture Flow
Client→ LB→ API Service→ Redis Cache→ PostgreSQL
POST /shorten→ hash(url)→ store mapping→ return short code
Key Decisions
Short code genBase62(counter) OR MD5(url)[0:7]. Base62 = no collisions, predictable. MD5 = collisions possible, need check
ID generationDistributed: Snowflake ID or DB auto-increment (single point). Use counter in ZooKeeper if multi-node
Redirect type301 (permanent, browser caches = less load) vs 302 (temporary, every req hits server = analytics data)
Read/Write ratio100:1 reads vs writes → optimize for reads. Cache hot URLs in Redis with TTL
Custom aliasesAllow user-defined slugs. Check uniqueness on write. Store in same table with flag
Schema & Scale
-- Core table urls(id BIGINT PK, short_code VARCHAR(8) UNIQUE, long_url TEXT, user_id, created_at, expires_at, click_count BIGINT DEFAULT 0) -- Indexes idx_short_code (short_code) ← most critical idx_user_id (user_id, created_at)
Scale Considerations
CachingCache shortCode→longURL in Redis. LRU eviction. Cache hit rate target: 80%+
DB scalingRead replicas for redirect lookups. Write to master only. Partition by short_code range if needed
AnalyticsAsync: publish click event to Kafka → consumer updates stats. Don't block redirect on analytics write
⚠ Collision handling: if MD5-based, check DB before insert. On collision, append counter or re-hash.

03
News Feed / Timeline (Twitter/Instagram)
Fanout strategies for social content delivery
Fanout Cache Queue ▶
Two Fanout Models
Fanout on WritePre-compute feed on post. Push to followers' Redis lists. Fast reads. Bad for celebrities (10M followers = 10M writes)
Fanout on ReadCompute feed at read time. No write amplification. Slow for active users following 1000s of accounts
Hybrid ✓Fanout-on-write for regular users. Fanout-on-read for celebrities (threshold: >N followers). Twitter uses this
Flow (Hybrid)
POST /tweet→ Write DB→ Kafka→ Fanout Worker
Fanout Worker→ Redis List→ per follower feed
GET /feed→ Redis→ merge celeb posts→ sort+return
Data Model
posts(id, user_id, content, media_url, created_at, like_count, comment_count) follows(follower_id, followee_id, created_at) -- Index: (follower_id) for "who do I follow?" -- Index: (followee_id) for "who follows me?" -- Redis feed (sorted set by timestamp) feed:{userId} → ZSET[(postId, timestamp)] -- Keep only latest 500-1000 posts in cache
Key Challenges
Hot usersCelebrity posts pulled separately at read time. Don't fanout Beyoncé's post to 100M feeds
Feed freshnessRedis list capped at N posts. Older posts fetched from DB on scroll (cursor pagination)
RankingChronological first. ML ranking: separate service that reranks top-N posts by engagement score

Chat System (WhatsApp / Slack)
Real-time messaging with persistence and delivery guarantees
WebSocket Queue Storage ▶
Architecture
Client A⇌ Chat Server→ Kafka
Kafka→ Delivery Service→ Chat Server B⇌ Client B
Protocol Choice
WebSocketBi-directional, persistent. Best for real-time chat. Client and server can both push messages
Long PollingFallback. Client holds open HTTP request until message arrives. Higher latency, simpler infra
SSEServer → Client only. Good for notifications, not for chat (need bidirectional)
Message States

Notification System
Push/email/SMS delivery at scale with reliability
Async Queue Retry ▶
Architecture
Trigger Event→ Notification Service→ Kafka
Kafka→ Email Worker→ SendGrid
Kafka→ Push Worker→ APNS / FCM
Kafka→ SMS Worker→ Twilio
Key Properties
IdempotencyEach notification has unique ID. Workers check DB before sending to prevent duplicates on retry
PrioritySeparate Kafka topics: critical (OTP, security alerts) vs normal (marketing). Critical = smaller, faster queue
PreferencesUser stores channel preferences. Check before sending: user.email_enabled, user.push_enabled
Retry & Reliability
Retry strategyExponential backoff: 1s, 2s, 4s, 8s... Max 5 retries. Dead letter queue for failures
Delivery trackingStore delivery status in DB: PENDING → SENT → DELIVERED/FAILED. Webhook from provider updates status
Rate limitingPer-user: max 10 notifications/hour. Per-provider: respect SendGrid/Twilio rate limits
Template engineSeparate template service. Supports i18n/localization. Store templates in DB, render at send time
Schema
notifications(id UUID, user_id, channel, template_id, payload JSONB, status, attempts INT, created_at, sent_at, error_msg)

Search Autocomplete / Typeahead
Fast prefix suggestions at <100ms p99
Trie Cache Ranking ▶
Data Structure Options
TriePrefix tree. O(p) lookup (p=prefix length). Store top-K suggestions at each node. Memory intensive
Redis ZSET ✓Store (suggestion, score) in sorted set. Range query by lex prefix. Simple, scalable, fast
Elasticsearchcompletion suggester. Best for complex use cases with NLP, fuzzy matching. Higher latency
Flow
User types "ap"→ Redis ZSET→ top 5 results
-- Redis sorted set with lex scoring ZADD suggestions 0 "apple" ZADD suggestions 0 "appetizer" ZADD suggestions 0 "app" -- Query prefix "ap" ZRANGEBYLEX suggestions "[ap" "[ap\xff" LIMIT 0 5 -- Score = -frequency (negative for desc order)
Ranking & Updates
ScoreCombine: search_frequency × recency_weight × personalization_boost
Update pipelineLog all searches to Kafka → Spark batch job computes top-K per prefix daily → update Redis
PersonalizationUser's recent searches stored in Redis. Merge global + personal results, deduplicate
FilteringBlock offensive suggestions. Maintain blocklist. Filter at serving layer, not storage layer
💡 Debounce on client side (300ms) to avoid hitting API on every keystroke. CDN can cache popular prefix results.
07
Payment System
Reliable, idempotent financial transaction processing
Consistency Idempotency ACID ▶
Flow
Client→ Payment API→ PSP
PSP→ Webhook→ Payment API→ DB update
Critical Properties
IdempotencyClient sends idempotency_key (UUID). Server checks if key already processed → return cached result. Never charge twice
ConsistencyUse DB transactions for: deduct balance + create transaction record atomically. NEVER partial updates
Audit logAppend-only ledger. Every state change recorded. Never UPDATE transactions, only INSERT new entries
ReconciliationDaily job: compare our DB vs PSP records. Flag discrepancies. Essential for financial compliance
Schema
payments(id UUID PK, idempotency_key VARCHAR UNIQUE, amount BIGINT, ← store in cents, not float! currency CHAR(3), status ENUM(PENDING,SUCCESS,FAILED), psp_reference VARCHAR, created_at, updated_at) ledger(id, payment_id, debit_account, credit_account, amount, created_at) ← immutable
Failure Handling
TimeoutPSP call timed out → status stays PENDING. Background job retries with same idempotency_key
Webhook retryPSP retries webhooks. Must be idempotent. Check status before processing duplicate webhook
RefundsCreate new REFUND transaction. Insert reversal entries in ledger. Call PSP refund API
⚠ NEVER store card numbers. Use PSP token (Stripe PaymentMethod). PCI-DSS compliance required.

Web Crawler / Search Indexer
Distributed crawling with dedup and politeness
Distributed Queue Bloom Filter ▶
Architecture
Seed URLs→ URL Queue→ Crawler Workers
Crawler→ Parse HTML→ Content Store
New URLs→ Bloom Filter→ URL Queue
Key Components
URL dedupBloom filter for fast probabilistic check. If "maybe seen" → check DB. Bloom filter = 1 bit per URL
Politenessrobots.txt compliance. Per-domain rate limiting. Respect Crawl-delay header. Never overload a single domain
Content dedupSimHash of page content. Similar hashes (hamming distance <3) = near-duplicate. Skip re-indexing
Priority queueHigher priority: sitemaps, fresh content, high PageRank pages. Lower priority: deep links, low traffic
Scale & Storage
URL frontierPriority queue backed by Kafka or Redis sorted set. Partition by domain for politeness control
Raw contentStore HTML in S3/object store. Store metadata (URL, crawl time, status) in DB
IndexingSeparate indexing pipeline: raw HTML → parser → tokenizer → inverted index update (Elasticsearch)
Schema
urls(id, url VARCHAR UNIQUE, status, priority INT, last_crawled_at, content_hash VARCHAR, domain) -- Bloom filter in Redis for fast dedup -- S3 for raw HTML content

Distributed Cache (Redis)
Caching strategies, patterns, and failure modes
Cache Patterns Failures ▶
Cache Strategies
StrategyWrite FlowUse When
Cache-asideApp writes DB, invalidates cache. Read: check cache → DB → populateMost common. Flexible. Cache only what's read
Write-throughApp writes cache AND DB synchronouslyRead-heavy. Ensures consistency. Double write latency
Write-behindApp writes cache. Async DB writeWrite-heavy. Risk: data loss if cache fails before DB write
Read-throughCache fetches from DB on miss automaticallySimplifies app code. Cache layer handles DB interaction
Failure Modes
⚠ Cache Avalanche: many keys expire simultaneously → DB overwhelmed. Fix: random jitter on TTL.
⚠ Cache Stampede: hot key expires → many concurrent requests hit DB. Fix: probabilistic early refresh or mutex lock on cache miss.
⚠ Cache Penetration: request for non-existent key always hits DB. Fix: cache null values with short TTL, or Bloom filter.
Redis Data Structures
StringSimple cache, counters, rate limiting. GET/SET. O(1)
HashObject storage (user profile fields). HGET/HSET. Efficient for partial updates
Sorted SetLeaderboards, feed ranking, rate limiting. ZADD/ZRANGE. O(log N)
ListMessage queues, activity feeds. LPUSH/RPOP. O(1) for head/tail
SetUnique visitors, tags, friend lists. SADD/SISMEMBER. O(1) average
Pub/SubReal-time notifications. Not persistent. If subscriber offline → message lost
Eviction Policies
allkeys-lru ✓ volatile-lru allkeys-lfu volatile-ttl noeviction
Use allkeys-lru for general cache. noeviction = returns error when full (bad for cache).

CDN Patterns
Push CDNYou upload content to CDN proactively. Good for small, rarely updated files.
Pull CDNCDN fetches from origin on cache miss. Auto-populates. Most common (Cloudflare, CloudFront).
Cache-Controlmax-age=86400 (1 day). stale-while-revalidate. CDN respects these headers.
InvalidationURL versioning (/v2/app.js) preferred over purge API (eventual, expensive).
What to CDNStatic assets, images, video segments, public API responses. NOT: auth endpoints, personalized data.
💡 CDN at edge = 50-300ms latency reduction. Cache hit rate 80-95% for well-configured static sites.
API Gateway
ResponsibilitiesAuth, Rate limiting, Request routing, SSL termination, Load balancing, API versioning
AggregationBFF (Backend-for-Frontend): gateway combines calls from multiple services. One client request → multiple service calls
Circuit BreakerCLOSED (normal) → OPEN (failing, fast-fail) → HALF-OPEN (probe). Prevents cascade failures.
Retry policyOnly retry idempotent ops (GET, PUT). Exponential backoff + jitter. Max 3 retries.
TimeoutAlways set. Downstream timeout < upstream timeout. Fail fast, don't queue forever.
Service Communication
PatternWhen
Sync REST/gRPCReal-time response needed. Query/response semantics.
Async EventsFire-and-forget. Decouple services. Long-running tasks.
Saga PatternDistributed transactions. Sequence of local transactions with compensating actions on failure.
Event SourcingAudit trail. Replay capability. State = sequence of events.
CQRSSeparate read/write models. Optimize reads independently. Adds complexity.
⚠ 2PC (Two-Phase Commit) = distributed ACID but blocking. Avoid in microservices. Use Saga instead.
ID Generation Strategies
StrategyProperties
UUID v4Random. No coordination needed. Not sortable. 36 chars. Index fragmentation.
UUID v7Time-ordered UUID. Sortable. Better index performance. Newer standard.
Snowflake64-bit: timestamp(41b) + datacenter(5b) + worker(5b) + seq(12b). Sortable, fast, distributed.
ULIDSnowflake alternative. URL-safe, monotonic within millisecond.
DB SequenceSimple. Single point. Good for single-DB setups. Not for distributed.
Snowflake: ~4096 IDs/ms/worker · 1023 workers · 69 years before overflow
Database Selection & Patterns
SQL vs NoSQL Decision
Use SQL WhenUse NoSQL When
Complex relationships, joins neededFlexible/evolving schema
ACID transactions requiredHorizontal scale needed
Financial data, strong consistencyHigh write throughput
Structured, normalized dataUnstructured / document data
Vertical scale (mostly)Eventual consistency acceptable
ACIDAtomicity, Consistency, Isolation, Durability. Guaranteed in SQL. Partial in some NoSQL.
CAP TheoremPick 2: Consistency, Availability, Partition Tolerance. P always happens → choose C or A.
CP systemsPostgreSQL, HBase, Zookeeper. Consistent but may be unavailable during partition.
AP systemsCassandra, DynamoDB, CouchDB. Always available, eventually consistent.

Database Scaling Patterns
VerticalBigger machine. Fast, no code change. Hard limit. SPOF. Do this first.
Read ReplicasAsync replication to replicas. Direct reads there. Replica lag = eventual consistency. Scale reads 10x.
ShardingPartition data horizontally across nodes. Shard key is critical. Bad key = hotspot. Cross-shard joins = hard.
Shard KeyChoose for even distribution. user_id (good) vs country (skewed). Hash sharding vs range sharding.
FederationSplit by function: users DB, orders DB, products DB. Removes single large DB bottleneck.
💡 Order: Caching → Read replicas → Vertical scale → Federation → Sharding
Indexing Strategy
B-Tree indexDefault. Range queries, equality, ORDER BY. Column cardinality matters. High cardinality = better.
Hash indexEquality only (=). Faster than B-Tree for exact match. No range queries.
CompositeMulti-column: (user_id, created_at). Left-prefix rule: (A,B,C) index helps A, AB, ABC — NOT B or C alone.
Partial indexIndex on subset: WHERE status='active'. Smaller, faster. Good for sparse conditions.
CostEvery index = slower writes (update index on insert/update/delete). Index maintenance overhead.
Covering indexInclude all SELECT columns in index. Query returns from index only. No table read needed.
⚠ Don't index low-cardinality columns (boolean, gender). Index selectivity = distinct/total. Target >0.1.
Database Consistency Patterns
Consistency Levels
StrongRead always returns latest write. SQL default. Higher latency.
EventualWrites propagate async. Reads may be stale. Higher availability. Cassandra default.
Read-your-writesYou always see your own writes. Others may be stale.
Monotonic readsWon't see older data after seeing newer. No time travel.
Transaction Isolation
Read UncommittedDirty reads. Never use.
Read CommittedNo dirty reads. PostgreSQL default. Non-repeatable reads possible.
Repeatable ReadSame query = same result within txn. Phantom reads possible.
SerializableFull isolation. Slowest. Use for critical financial ops.
Replication Modes
Sync replicationWrite ACKed only after replica confirms. Strong consistency. Higher write latency.
Async replicationWrite ACKed immediately. Replica may lag. Risk: data loss if master fails before replica syncs.
Semi-syncWait for at least 1 replica. Balance: durability vs latency. MySQL semi-sync.
Quorum writesWrite to W of N nodes. W + R > N = strong consistency. Cassandra tunable.

Push vs Pull Architecture
PushServer sends data to client. WebSocket, SSE, webhooks. Lower latency. Harder to scale (maintain connections).
Pull (Polling)Client asks server periodically. Simple. Higher latency. Wasteful if no new data. Use Long Polling as middle ground.
Long PollingClient holds open connection until event or timeout. Then immediately re-connects. Simpler than WebSocket.
SSEServer → Client streaming over HTTP. One-way. Auto-reconnect built in. Good for dashboards, live feeds.
WebSocketBi-directional. Persistent. Best for chat, gaming, collaboration. Sticky sessions needed at LB layer.
Scalability Escalation Path
1. Add caching (Redis) → solve 80% of problems 2. Add CDN → offload static assets + public API 3. Read replicas → scale read-heavy workloads 4. Vertical scaling → buy time before horizontal changes 5. Async processing (queues) → decouple slow operations 6. Horizontal scaling (more instances) → stateless services only 7. Database federation → split by domain/function 8. Sharding → last resort, high complexity 9. Microservices → when teams, not load, is the limit
💡 In interviews: always go in this order. Don't jump to sharding for a problem caching solves.
